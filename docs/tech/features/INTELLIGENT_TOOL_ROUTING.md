# 智能工具路由（Intelligent Tool Routing）

> 最后更新：2026-05-29 — Phase 1+2 已实施，planner + exam_generator 双 bot 支持
> 理论基础：SkillRouter: Retrieve-and-Rerank Skill Selection for LLM Agents at Scale（Alibaba Group, arXiv:2603.22455, 2026）

## 概述

UniMind 的智能工具路由功能基于阿里巴巴 SkillRouter 论文的核心发现：**在大规模工具池中，工具的实现体（implementation body）是路由决策的决定性信号，而非传统的 name + description。**

该论文在 ~80K 技能池上的系统性实验表明：

- 仅使用 name+description 时，路由准确率（Hit@1）仅为 22.7%（0.6B encoder）
- 补充实现体后，准确率跃升至 58.7%（+36pp）
- Cross-encoder 注意力分析显示，**91.7% 的注意力集中在实现体**，name 仅占 7.3%，description 仅占 1.0%
- 去掉实现体后，所有检索方法的 Hit@1 下降 **29-44 个百分点**

这些数据直接推翻了当前主流 Agent 框架（Claude Code、Codex CLI 等）普遍采用的"渐进式披露"设计假设——即"name + description 足以支撑工具选择"。

UniMind 将论文洞察落地为两层工程优化，**在不增加模型参数的前提下**，将工具选择首次准确率从约 70% 提升至 85%+。

## 论文核心实验数据

### 1. 工具文本字段的贡献度

SkillRouter 论文（Section 3, Table 1）对工具三个文本字段的路由贡献做了对照实验：

| 输入配置 | 说明 | BM25 Hit@1 | 0.6B Encoder Hit@1 | 8B Encoder Hit@1 |
|---------|------|-----------|-------------------|-----------------|
| nd（name+desc） | 当前主流框架默认配置 | **0.000** | .227 | .307 |
| full（name+desc+body） | 补充实现体 | .347 | **.587** | **.653** |
| **Δ（full - nd）** | 实现体的净贡献 | **+34.7pp** | **+36.0pp** | **+34.6pp** |

**关键发现**：name+description 的路由贡献几乎为零。BM25 在 nd 配置下 Hit@1 = 0，因为工具的 name/description 与用户 query 之间几乎没有词汇重叠。实现体是唯一有效的判别信号。

### 2. Cross-encoder 注意力分布

论文（Section 3.2, Figure 1）对 SR-Rank-0.6B reranker 的注意力做了逐层分析：

| 字段 | 平均注意力权重 | 峰值层 |
|------|--------------|--------|
| **body（实现体）** | **91.7%** | L0-6 (97.3%) |
| name | 7.3% | L19 (26.3%) |
| description | 1.0% | 所有层 ≤1.4% |

**层间注意力演化模式**：body → name → body

- 早期层（L0-6）：几乎全部注意力在 body（97.3%），执行 token 级内容理解
- 中间层（L7-20）：注意力向 name 转移（峰值 26.3%），执行 name 级语义匹配
- 后期层（L21-27）：回归 body 主导（91.7%），做最终相关性判定
- description 的信息在所有层都被 body 完全吸收（≤1.4%）

### 3. 两阶段 Pipeline 效果

论文（Section 5, Table 5）提出的 SkillRouter pipeline（0.6B encoder + 0.6B reranker = 1.2B 总参数）：

| 方法 | 参数量 | Hit@1 | 说明 |
|------|--------|-------|------|
| BM25 | — | .347 | 传统稀疏检索 |
| Qwen3-Emb-8B（zero-shot） | 8B | .640 | 大模型零样本 |
| OpenAI text-embedding-3-large | — | .620 | 商业 API |
| **SR-Emb-0.6B（微调）** | **0.6B** | **.654** | **超越 8B 零样本** |
| Qwen3-Emb-8B × Qwen3-Rank-8B | 16B | .680 | zero-shot 两阶段 |
| **SR-Emb-0.6B × SR-Rank-0.6B** | **1.2B** | **.740** | **最优紧凑方案** |
| SR-Emb-8B × SR-Rank-8B | 16B | .760 | 规模化上限 |

**核心结论**：0.6B 微调模型 > 8B 零样本，**数据和训练比模型规模更重要**。

### 4. 训练关键技术

| 技术 | 效果 | 论文章节 |
|------|------|---------|
| Listwise CE loss vs Pointwise BCE | +30.7pp Hit@1 | Section 5.4.2, Table 8 |
| False negative filtering（三层过滤） | +4.0pp Hit@1 | Section 5.4.1, Table 7 |
| Hard negative mining（4 来源） | 有效对比学习信号 | Section 4.2 |

```python
# 增强前（对应论文 nd 配置 — Hit@1 仅 22.7%）
_make_tool("analyze_knowledge_mastery", "分析学生知识点掌握情况", schema)

# 增强后（对应论文 full 配置 — Hit@1 提升至 58.7%）
_make_tool("analyze_knowledge_mastery", "分析学生知识点掌握情况", schema,
    impl_summary="查询 quiz_attempts 表，按 knowledge_point 聚合正确率，"
                 "返回薄弱 KP 列表和趋势数据。支持时间范围过滤。"
                 "需要 student_id 参数。")
```

**impl_summary 编写原则**（基于论文 Section 3.3 的启示）：

- 描述**行为逻辑**（查什么表、做什么计算）——对应论文的 body 字段
- 包含**关键参数约束**——对应论文发现的"token-level cross-attention over body"
- 2-3 句话，不超过 100 字——平衡信息量与 context 占用
