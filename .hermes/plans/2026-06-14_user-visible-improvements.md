# 用户体验提升计划

**目标**：把基建投入转化为用户能看见、能感受到的价值。从"引擎很好但没有油"变成"学生提分看得见、教师省力感明显、机构上手就能用"。

**创建时间**：2026-06-14

---

## 一、现状

### 我们有什么

- 37 个前端页面、11 个 Django app
- 两个自治 Agent（小宇、工作台），完整的工具调用和意图路由
- Memorix 记忆调度、IRT 参数估计、ARC 对抗出题管线、GEPA 自进化
- 11 个学科知识树（结构完整）
- 多租户 RBAC、支付网关、OSS 存储

### 用户实际能看到什么

- **学生**：登录 → 诊断测试 → 小宇对话 → 刷题。200 道种子题，刷 10 分钟见底。课程/视频内容为零。
- **教师**：工作台对话出题 → 手动查看结果。没有组卷、布置、批改闭环。
- **机构管理员**：空白页面。不知道怎么开始。

### 核心矛盾

**基建自洽，内容空虚。** 200 道题 ÷ 11 个学科 = 每科不到 20 道。所有算法、Agent、Pipeline 在内容不足的情况下都是空转。

---

## 二、不做什么

明确定界，防止 scope creep：

- **不加新算法/新 Pipeline**。Memorix-Field、Experience Router、GEPA 都够前沿了。
- **不做移动端 App**。内容没填满之前 PWA 够用。
- **不做社交/排行榜/gamification**。没内容时做这些是给空房子贴墙纸。
- **不碰 Agent 人格/对话体验调优**。那不是当前的瓶颈。

---

## 三、做什么（四个工作流，按优先级）

### 工作流 1：内容生产——ARC 管线批量出题

**优先级**：最高。这是所有其他工作的前提。

**做什么**：
基于已有 ARC Pipeline（Author → Reviewer → AuthorRevise → Classifier），构建**批量模式**。输入知识树节点，一口气生成 50-100 道题，自动走完质检管线，合格的入库，不合格的丢弃。

**目标**：
- 每学科 ≥ 500 道可用题目（当前：平均 18 道/学科）
- 客观题占比从 2% 提升到 ≥ 40%
- 难度分布均匀（easy:normal:hard ≈ 1:2:1）
- 入库通过率 ≥ 70%

**用户可见的产出**：
- 学生刷题不再"见底"
- 教师组卷有题可选
- 知识图谱每个节点下有对应题目
- 诊断测试题目不再重复

**边界**：
- 不追求 100% 通过率。70% 就够，剩下的丢弃。
- 不使用 thinking 模式（成本太高），用 fast 模型。
- 每批生成控制在 3-5 并发（已有 `AI_BULK_GENERATE_CONCURRENCY=4`）。

### 工作流 2：学生端——让提分看得见

**优先级**：第二。内容有了之后，学生需要感知到进步。

**做什么**：

**2a. StudentHome 改造为 Progress Dashboard**

当前 StudentHome 本质是学习计划查看器。需要改成真正的仪表盘：

| 模块 | 内容 | 数据来源 |
|------|------|----------|
| 本周掌握 | +X 个知识点，-Y 个退步 | Memorix 掌握度变化 |
| 连续学习 | 连续 N 天，今日目标完成度 | 答题记录 |
| 薄弱知识点 Top 3 | 正确率最低的三个节点 | 答题统计 |
| 小宇提醒 | 今天该复习什么 | Memorix 调度 + Agent 生成 |
| 预计得分趋势 | 近 30 天 ELO 变化曲线 | ELO 历史 |

这些数据**全都有**，只是没有呈现在一个页面上。

**2b. 知识图谱叠加个人掌握度**

KnowledgeMap 页面已用 D3.js 渲染知识拓扑，在此基础上叠加热力图：
- 绿色节点 = 掌握（正确率 ≥ 80%）
- 黄色节点 = 一般（50%-80%）
- 红色节点 = 薄弱（< 50%）
- 灰色节点 = 未涉及

改动量不大（前端已有渲染管线），效果直接——学生一眼看到自己哪里需要补。

### 工作流 3：教师端——从"能出题"到"能上课"

**优先级**：第三。内容 + 学生可见之后，教师需要完整工作流。

**做什么**：

**3a. 组卷（Paper Composition）**

在工作台的出题结果上增加：
- 选题 → 调整顺序 → 预览试卷
- 生成 PDF（试卷 + 答题卡，已有 `PdfMockExam` 的 PDF 生成能力可复用）
- 保存为模板（已有 `ExamTemplate` model）

**3b. 一键布置 → 自动批改 → 结果总览**

三步串成一条线：
1. 选择试卷模板 → 选择班级 → 设定截止时间 → 发布
2. 学生提交后：客观题自动判分，主观题 AI 判分（已有主观题判分引擎）
3. 教师端查看：全班正确率分布、每题正确率、需要关注的学生名单

这不是新能力，是把已有能力（出题、判分、Assignment model、班级管理）串成工作流。

### 工作流 4：机构 Onboarding

**优先级**：第四。降低新机构的流失率。

**做什么**：

**4a. Quick Start Wizard**

新机构管理员首次登录 → 三步引导：
1. 选择学科 → 自动灌入知识树 + 500 道种子题
2. 导入学生（CSV 上传 或 批量生成邀请链接）
3. 预览效果 → 完成

目标：机构 5 分钟内从空白到可用。

**4b. 内容导入工具**

教师手里有现成的试卷/题库（Word/Excel），提供拖拽上传 → AI 解析 → 自动入库。这是 B 端销售的关键能力——"你们现有的题目可以直接导入"。

---

## 四、时间建议

不拆具体工期（那是"怎么做"），只说先后顺序：

1. **先做工作流 1**（内容生产）。没有内容，2/3/4 都是空中楼阁。预计 1-2 周让管线跑通并产出第一批题目。
2. **2a 和 2b 可以并行于 1 的后期**。数据都在，纯前端改造。
3. **工作流 3 在内容量达标后启动**。组卷需要足够的题目池。
4. **工作流 4 随时可以做**。Quick Start Wizard 不依赖内容量（用 AI 生成的种子数据即可），但需要机构有实际题目后才能验证导入工具。

---

## 五、风险和开放问题

- **ARC 批量生成的 API 成本**：每道题 4 次 LLM 调用（Author/Reviewer/AuthorRevise/Classifier），每学科 500 道 = 2000 次调用/学科。需要先跑一个学科验证成本和质量。
- **批量生成的质量稳定性**：ARC 单题精修质量高，批量模式可能下降。需要设定最低通过标准并严格执行丢弃。
- **内容导入工具的复杂度**：Word/Excel 格式千差万别，AI 解析的准确率需要实测。建议先支持 CSV（结构化），再扩展到 Word。
- **Progress Dashboard 的数据新鲜度**：Memorix 掌握度是实时计算的还是缓存的？如果每次加载页面都要重算可能慢。
